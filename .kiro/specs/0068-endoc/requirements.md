# 英語ドキュメントの用意の要件定義書

## 1. 概要

### 1.1 プロジェクト情報
- **プロジェクト名**: go-webdb-template
- **Feature名**: 0068-endoc
- **作成日**: 2026-01-27
- **関連Issue**: https://github.com/taku-o/go-webdb-template/issues/140

### 1.2 目的
プロジェクトに英語版のドキュメントを用意し、日本語と英語の両方でドキュメントを提供できるようにする。日英で差分が生じた場合のマスタードキュメントは日本語版とし、内部コメントは従来通り日本語を使用する。

### 1.3 スコープ
- README.mdの英語版作成（既存のREADME.mdはREADME.ja.mdにリネーム）
- docsディレクトリ内の全Markdownファイルの英語版作成（既存ファイルはdocs/ja/に移動）
- docs/_config.ymlのdescriptionを英語に変更

**本実装の範囲外**:
- ドキュメントの内容の大幅な変更・追加
- 既存のドキュメント構造の変更
- コード内のコメントの英語化
- その他の設定ファイルの英語化

## 2. 背景・現状分析

### 2.1 現状
- README.mdは日本語で記載されている
- docsディレクトリ内の全ドキュメント（24ファイル）が日本語で記載されている
- docs/_config.ymlのdescriptionが日本語で記載されている
- 英語版のドキュメントが存在しない

### 2.2 問題点
- 英語話者や国際的なコントリビューターがプロジェクトを理解しにくい
- プロジェクトの国際的な展開や採用が困難
- 英語圏でのプロジェクトの認知度向上が困難

### 2.3 必要性
- プロジェクトの国際的な展開を支援する
- 英語話者によるコントリビューションを促進する
- プロジェクトの認知度を向上させる
- 多言語対応により、より多くの開発者にアクセスできるようにする

### 2.4 実現可否
- 既存のドキュメントを翻訳することで対応可能
- ファイルの移動とリネームは技術的に容易
- 既存のドキュメント構造を維持しながら英語版を追加できる

## 3. 機能要件

### 3.1 README.mdの英語化

#### 3.1.1 ファイルのリネーム
- **変更前**: `README.md`（日本語版）
- **変更後**: `README.ja.md`（日本語版）
- **新規作成**: `README.md`（英語版）

#### 3.1.2 翻訳要件
- 既存のREADME.mdの内容を英語に翻訳
- 構造、セクション、コード例は維持
- リンクや参照先は適切に更新
- 冒頭または末尾に言語切替リンクを追加: `**[日本語] | [English]**`

### 3.2 docsディレクトリの英語化

#### 3.2.1 既存ファイルの移動
- **変更前**: `docs/*.md`（24ファイル、日本語版）

#### 3.2.2 英語版ファイルの作成
- **新規作成**: `docs/en/*.md`（24ファイル、英語版）
- 各ファイルの内容を英語に翻訳
- ファイル名は英語版でも同じ名前を維持（必要に応じて調整）
- 各ファイル（日本語版と英語版）の冒頭または末尾に言語切替リンクを追加: `**[日本語] | [English]**`

### 3.3 docs/_config.ymlの英語化

#### 3.3.1 descriptionの変更
- **変更前**: `description: "大量ユーザー、大量アクセスの運用に耐えうるGo APIサーバー、データベース構成のテンプレートプロジェクト"`
- **変更後**: `description: "Template project for Go API server and database configuration that can withstand large-scale users and high traffic operations"`

### 3.4 ドキュメントの整合性

#### 3.4.1 マスタードキュメント
- 日英で差分が生じた場合のマスタードキュメントは日本語版とする
- 日本語版を優先的に更新し、英語版はそれに追従する

#### 3.4.2 内部コメント
- コード内のコメントは従来通り日本語を使用
- ドキュメントの英語化はコード内コメントには影響しない

## 4. 非機能要件

### 4.1 互換性
- 既存のドキュメントリンクが機能し続けること（必要に応じてリダイレクトや更新）
- GitHub Pagesなどの既存の公開方法が機能し続けること
- 既存のドキュメント参照が壊れないこと

### 4.2 保守性
- 日本語版と英語版の両方を維持できる構造にする
- 将来的に他の言語を追加しやすい構造にする
- ドキュメントの更新が容易な構造にする

### 4.3 品質
- 英語版の翻訳品質を確保する
- 技術用語の一貫性を保つ
- コード例やコマンド例が正しく動作することを確認する

### 4.4 パフォーマンス
- ドキュメントの追加によるビルド時間や公開時間への影響を最小限にする
- ファイルサイズの増加を考慮する

## 5. 制約事項

### 5.1 技術的制約
- 既存のドキュメント構造（Markdown形式）を維持する
- GitHub PagesやJekyllの設定との互換性を保つ
- 既存のリンク構造を維持する

### 5.2 実装上の制約
- コード内のコメントは日本語のまま維持する
- 既存のドキュメントの大幅な構造変更は行わない
- 翻訳は既存の内容を基に行う

### 5.3 機能の制約
- 本実装は英語版の追加のみを対象とする
- 他の言語への対応は範囲外とする
- ドキュメントの自動翻訳機能は範囲外とする

## 7. 影響範囲

### 7.1 変更されるファイル
- `README.md`: 英語版に置き換え
- `README.ja.md`: 新規作成（既存のREADME.mdからリネーム）
- `docs/*.md`: `docs/ja/*.md`に移動（24ファイル）
- `docs/en/*.md`: 新規作成（24ファイル）
- `docs/_config.yml`: descriptionを英語に変更

### 7.2 既存ファイルへの影響
- コードファイルへの影響はない
- 設定ファイル（_config.yml以外）への影響はない
- テストファイルへの影響はない

### 7.3 既存機能への影響
- 既存のドキュメント参照リンクが更新される必要がある場合がある
- GitHub Pagesの設定が更新される必要がある場合がある
- 既存のドキュメントビルドプロセスへの影響を確認する必要がある

## 8. 実装上の注意事項

### 8.1 ファイル移動の順序
1. 既存のREADME.mdをREADME.ja.mdにリネーム
2. docsディレクトリ内の全Markdownファイルをdocs/ja/に移動
3. 英語版のREADME.mdを作成
4. 英語版のdocs/en/*.mdを作成
5. docs/_config.ymlを更新
6. 各Markdownファイル（README.md、README.ja.md、docs/ja/*.md、docs/en/*.md）の冒頭または末尾に言語切替リンクを追加

### 8.2 リンクの更新
- ドキュメント内の相互参照リンクを適切に更新する
- 相対パスや絶対パスが正しく機能することを確認する
- GitHub PagesのURL構造を考慮する
- 言語切替リンクのパスが正しいことを確認する
  - README.md（英語版）: `**[日本語](README.ja.md) | [English]**`
  - README.ja.md（日本語版）: `**[日本語]** | [English](README.md)`
  - docs/ja/*.md（日本語版）: `**[日本語]** | [English](../en/{filename})`
  - docs/en/*.md（英語版）: `**[日本語](../ja/{filename}) | [English]**`

### 8.3 翻訳の品質
- 技術用語の一貫性を保つ
- コード例やコマンド例はそのまま維持する
- 文脈を理解した上で翻訳する

### 8.4 テスト
- 各ドキュメントファイルが正常に表示されることを確認する
- リンクが正しく機能することを確認する
- 言語切替リンクが正しく機能することを確認する
- ビルドエラーが発生しないことを確認する
